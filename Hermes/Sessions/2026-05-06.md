## Trabajo Técnico Realizado

### 1. Root Cause Analysis — MiniMax model_dump crash

**Problema:** 12 de 14 cron jobs fallando desde ~10:45 del 05/05.

**Root cause confirmada:** MiniMax API devuelve `reasoning_details` que contienen objetos dict-subclass con un `model_dump()` que falla. En `run_agent.py` línea 8886, `isinstance(d, dict)` retornaba True (correcto), pero estos son dict-subclasses que la SDK wrapping expone como dict-like objects. El elif `hasattr(d, "model_dump")` en línea 8890 se ejecutaba porque la dict-subclass tiene ese atributo, y `d.model_dump()` crasheaba.

Error exacto: `'dict' object has no attribute 'model_dump'` — la dict-subclass se imprime como dict en el mensaje de error.

**Fix 1 — YA APLICADO:** Convertido `Hermes Auto-Solve — Every 15min` (8af6ce471577) a `no_agent` mode. Ya no pasa por el AI pipeline — corre como subprocess Python directo. Verificado: el script corre clean sin error.

**Fix 2 — YA APLICADO:** Patch en `run_agent.py` líneas 8883-8893. Reordenado el check: `hasattr(d, "model_dump") and not isinstance(d, dict)` primero, luego dict, luego __dict__. Esto protege contra dict-subclasses con model_dump roto.

**Fix 3 — REQUIERE JUAN:** El gateway y scheduler de hermes-agent (PID 147853, 162257) necesitan restart para pick up del fix de código en `run_agent.py`. Comando: `hermes update && hermes gateway restart`.
